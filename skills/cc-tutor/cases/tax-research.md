# 사례: 세법리서치 스킬 — MCP 3종 + 3-Agent 병렬 + "1.5단계 재검증"으로 환각을 거른다

> 배우는 패턴: (e) MCP 활용 + 3-Agent 병렬 + 1.5단계 재검증  |  난이도: ★★★★★

## 1. 이 스킬이 푸는 실무 문제

세법 자문은 예규·판례 번호가 단 하나라도 틀리면 신뢰가 무너진다. 그런데 LLM은 그럴듯한 가짜 판례번호를 지어내기 쉽다. 이 스킬은 세 개의 서로 다른 소스(NotebookLM, 법제처 API MCP, 웹/정부사이트)를 **동시에** 돌려 같은 쟁점을 교차 조사한 뒤, 메인 루프가 직접 법제처 API로 "이 번호 진짜 있냐, 날짜·결론 맞냐"를 다시 확인(1.5단계)하고, 통과한 것만 노션 자문기록 DB에 남긴다. 정확성을 분량보다 우선하는 구조다.

## 2. SKILL.md 핵심 발췌

frontmatter가 아키텍처와 버전 이력을 압축한다 — v2.0의 핵심은 "1.5단계 신설".

```yaml
---
name: 세법리서치
description: "세법 쟁점을 NotebookLM + korean-law MCP + 웹 검색 3개 Agent
  병렬 리서치 후, 메인 루프가 A·C에서 나온 예규/판례 번호를 korean-law(B)로
  1.5단계 일괄 재검증한 뒤 노션 자문기록 DB에 기록. /세법리서치 로 호출."
version: "2.0"
---
```

아키텍처가 오케스트레이터(메인 루프) + 3 Agent + 재검증 게이트로 그려진다.

```markdown
[메인 루프 = 오케스트레이터]
 |-- 0단계: 쟁점 분석, 분야 판별, 거래처 확인
 |-- 1단계: 3개 Agent 동시 호출 (단일 메시지, 병렬)
 |     |-- Agent A: NotebookLM 리서처
 |     |-- Agent B: korean-law MCP 리서처 (법제처 API = 1차 신뢰 소스)
 |     |-- Agent C: WebSearch + Playwright 보완 리서처
 |-- 1.5단계 [NEW]: A·C가 발굴한 번호 중 B가 못 잡은 것만
 |                  korean-law로 ID·날짜·결론·키워드 4항목 재대조
 |-- 2단계: 교차검증 마커 확정 — ✓ / ◐ / △ / ✗
 |-- 3단계: 답변 작성  |-- 4단계: 노션 자문기록 등록
```

1.5단계의 4항목 대조와 마커 부여 규칙이 환각 필터의 본체다.

```markdown
### 1.5단계: 예규/판례 번호 일괄 재검증 (메인 루프)
대상: A·C가 인용한 번호 중 B가 이미 회수한 것은 제외 (중복 검증 안 함)
각 건: search_decisions로 ID 매칭 → 핵심 건만 get_decision_text 본문 회수
4항목 대조: 문서번호 / 날짜 / 결론(인용·기각, 과세·비과세) / 본문 키워드
 - 4개 모두 일치 → ✓ 교차확인
 - 1~3개만 일치 → ◐ 부분확인 (어느 항목이 다른지 비고에 한 줄)
 - 검색 0건  → △ 단일소스 ("korean-law 미확인" 표기)
 - 결론·날짜 다름 → ✗ 인용 금지, 표에서 제외
```

NotebookLM 노트북 ID는 매번 조회하지 않고 캐시 파일로 기억한다(파일에 분야→ID 매핑을 보관). MCP 비동기 조회는 `query_start`로 한 번에 큐잉하고 `query_status`를 백오프(0→20→50→110→170초)로 폴링한다.

## 3. 구조

```
skills/세법리서치/
├── SKILL.md                  # 오케스트레이션 + 3 Agent 프롬프트 템플릿
├── notebook_cache.json       # 분야→노트북ID 매핑 캐시 (ID는 파일이 기억)
├── owner_cache.json          # 담당자 노션 page URL 캐시
└── _archive/SKILL_v1.0.md    # 이전 버전 백업 (롤백용)

사용 MCP: NotebookLM / korean-law(법제처 Open API) / WebSearch·Playwright / 노션

흐름:
질문 → 0.쟁점분석 → 1.[A‖B‖C 동시] → 1.5.번호 재검증(메인이 직접 B호출)
     → 2.마커확정(✓◐△✗) → 3.답변 → 4.노션 기록 → 5.보고
```

## 4. 여기서 배울 패턴 3가지

1. **메인 루프가 오케스트레이터, 서브 Agent는 수집기.** 메인 루프가 PM+파트너 역할을 직접 쥐고, 3개 Agent는 각자 소스에서 자료만 긁어 구조화 JSON으로 반환한다. 메인이 세 결과 원본을 모두 보유하므로 정보 손실 없이 교차검증할 수 있다 — 판단은 중앙에서, 수집은 분산으로.

2. **소스마다 신뢰도를 다르게 두고, "1.5단계"라는 별도 검증 게이트를 둔다.** 법제처 API(B)를 1차 신뢰 소스로 삼되 ground truth로 맹신하진 않고, A·C가 새로 발굴한 번호는 B로 다시 돌려 4항목을 대조한다. ✓/◐/△/✗ 마커로 신뢰도를 답변 표에 그대로 노출해, 독자가 어떤 건을 검증된 것으로 볼지 알 수 있다.

3. **병렬 호출과 캐시로 비용·지연을 관리한다.** 3 Agent는 반드시 단일 메시지에서 동시 호출(순차 금지)하고, NotebookLM ID는 캐시 파일로 기억해 매번 list 호출을 피한다. 느린 RAG는 동기 호출 대신 비동기 큐잉+백오프 폴링으로 타임아웃을 견딘다. 다중 소스 리서치의 현실적 운영 기법이 모두 담겼다.

## 5. 토론 질문

1. 여러분 분야에서 "여러 소스가 서로 어긋날 수 있는" 사실은 무엇인가요? 그걸 교차검증한다면 어느 소스를 1차 신뢰로, 어느 소스를 "재확인 대상"으로 둘까요?
2. 이 스킬은 검증 결과를 ✓/◐/△/✗ 마커로 답변에 그대로 보여줍니다. 결과물에 "신뢰도 등급"을 노출하는 것의 장점과, 반대로 사용자를 혼란스럽게 할 위험은 각각 무엇일까요?
