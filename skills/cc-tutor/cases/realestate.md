# 사례: 부동산자문 스킬 — subagent 6개를 2파동으로 병렬 오케스트레이션하다

> 배우는 패턴: (f) subagent 병렬 오케스트레이션 (Agent A/B/C → D/E/F)  |  난이도: ★★★★★

## 1. 이 스킬이 푸는 실무 문제

서울 25개구 아파트 실거래로 개인 매입 후보 단지를 추리려면, 구마다·월마다 공공데이터 API를 수백 번 호출하고, 단지 상세·좌표·전세까지 따로 긁어 합쳐야 한다. 한 에이전트가 순차로 돌면 너무 느리다. 이 스킬은 작업을 권역·도메인별로 쪼개 **subagent를 두 파동으로 병렬** 돌린다 — 1파동(A/B/C)은 매매 실거래를 권역별로, 2파동(D/E/F)은 단지상세·좌표·전세를 도메인별로 동시 수집한다. 메인 루프는 수집 결과를 합쳐 스코어링하고 Top 10 표를 낸다.

## 2. SKILL.md 핵심 발췌

frontmatter는 입력·산출물을 한 줄로 정의한다(노션 저장 없이 대화창 출력만).

```yaml
---
name: 부동산자문
description: "서울 25개구 아파트 실거래 기반 개인 부동산 매입 분석.
  씨드·목적 입력받아 예산필터 + 전세가율/갭 + 역세권 + 단지규모 +
  5년 수익률로 Top 10 단지 표 제시. /부동산자문 로 호출."
---
```

아키텍처가 "메인 루프 + 2파동 × 3 Agent" 구조를 그대로 보여준다.

```markdown
[메인 루프 = 오케스트레이터]
 |-- Phase 0: 입력 수집 (씨드/목적/연소득/보유주택수)
 |-- Phase 1: 예산 범위 산정 (LTV/DSR)
 |-- Phase 2: 매매 실거래 병렬 수집 — 3 Agent 동시
 |     |-- Agent A (강남권 7개구)
 |     |-- Agent B (강북·도심 10개구)
 |     |-- Agent C (서남·서북 8개구)
 |-- Phase 2.5: 구별 5년 CAGR 산출 (메인 루프)
 |-- Phase 3: 단지 그룹핑 + 예산 필터 (메인 루프)
 |-- Phase 4+5: 단지상세 + 좌표 + 전세 병렬 수집 — 3 Agent 동시
 |     |-- Agent D: 단지목록 + 단지상세 (kaptCode 매칭)
 |     |-- Agent E: 지하철역 + 지오코딩 (역세권 거리)
 |     |-- Agent F: 전세 실거래 (전세가율/갭)
 |-- Phase 6: 스코어링 + 5년 수익률  |-- Phase 7: Top 10 표 출력
```

각 Agent는 "할당 범위의 MCP 호출 + JSON 구조화"만 담당하고, 반환 포맷이 못 박혀 있다.

```markdown
메인 루프가 PM+애널리스트 역할을 직접 수행. 서브 Agent는 수집·구조화만.

Agent A/B/C 반환 (공통):
[{"aptNm":"...","umdNm":"...","sigungu_code":"11710","area_m2":84.8,
  "amount_만원":250000,"date":"2026-03-12","period":"recent"}, ...]
- period: 최근 3개월="recent", 5년 전 3개월="past" (CAGR 산출용)
- "원본 텍스트 출력 금지. 반드시 위 JSON 배열만 반환."
```

스코어는 6항목 가중합(가격25%·거래량15%·전세가율15%·단지규모15%·역세권20%·추세10%)으로 메인 루프가 계산한다.

## 3. 구조

```
skills/부동산자문/
└── SKILL.md   # 7 Phase 오케스트레이션 + 6개 Agent 프롬프트 템플릿
                (별도 모듈 없이 메인 루프가 계산까지 직접 수행)

사용 MCP: public-data-kr (실거래·단지·지하철·지오코딩·공시가)

2파동 병렬:
[1파동] Phase2  A(강남7) ‖ B(강북10) ‖ C(서남8)  → 매매 실거래
          ↓ 메인: CAGR 산출 → 단지 그룹핑 → 예산필터 → Top 50
[2파동] Phase4+5  D(단지상세) ‖ E(좌표·역세권) ‖ F(전세)  → Top50 보강
          ↓ 메인: 스코어링 + 5년수익률 → Top 10 표 + 투자의견
```

## 4. 여기서 배울 패턴 3가지

1. **작업을 "차원"으로 쪼개 병렬화한다.** 1파동은 지리적 권역(강남/강북/서남)으로, 2파동은 데이터 도메인(상세/좌표/전세)으로 나눈다. 같은 종류의 호출을 권역별로 나누면 부하가 고르게 퍼지고, 서로 독립이라 동시에 돌려도 충돌이 없다. "반드시 단일 메시지에서 동시 호출"이 병렬성의 전제다.

2. **2파동 사이에 메인 루프의 수렴(reduce) 단계를 둔다.** 1파동 결과만으로는 어떤 단지를 깊게 볼지 모른다. 메인이 CAGR을 산출하고 예산 필터로 Top 50을 추린 뒤에야 2파동이 "그 50개만" 상세 조회한다. map(병렬 수집) → reduce(중앙 집계) → map(다시 병렬) 구조로, 2파동의 호출 수를 50개로 줄인다.

3. **서브 Agent의 출력 계약을 엄격히 고정한다.** 각 Agent는 정해진 JSON 스키마만 반환하고 "원본 텍스트 출력 금지"가 명시돼 있다. 판단·스코어링은 전부 메인 루프가 하고 Agent는 수집기에 머문다. 계약을 좁히면 메인이 6개 결과를 손실 없이 합칠 수 있다.

## 5. 토론 질문

1. 여러분 업무에 "독립적으로 동시에 처리할 수 있는데 지금은 순차로 하는" 작업이 있나요? 그걸 권역·카테고리 같은 차원으로 쪼갠다면 몇 개 subagent로 나눌 수 있을까요?
2. 이 스킬은 모든 점수 계산을 서브 Agent가 아니라 메인 루프에 모았습니다. 만약 각 Agent가 스스로 점수까지 매겨 반환하게 했다면 어떤 문제가 생길까요? "수집은 분산, 판단은 중앙" 원칙의 장단점은?
