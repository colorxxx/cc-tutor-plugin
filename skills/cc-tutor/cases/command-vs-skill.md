# 사례: 세법리서치 command vs skill — 슬래시 명령은 "얇은 진입점", 스킬은 "두꺼운 본체"

> 배우는 패턴: (g) slash command vs skill 차이  |  난이도: ★★

## 1. 이 스킬이 푸는 실무 문제

같은 "세법리서치" 기능에 두 개의 파일이 존재한다 — `commands/세법리서치.md`(슬래시 명령)와 `skills/세법리서치/SKILL.md`(스킬). 둘은 경쟁 관계가 아니라 역할 분담이다. command는 사용자가 `/세법리서치 <질문>`을 칠 때의 **얇은 진입점**이고, 실제 워크플로(3 Agent 병렬, 1.5단계 재검증 등)는 SKILL.md라는 **두꺼운 본체**에 들어 있다. command는 본체를 "읽고 그대로 따르라"고 지시한 뒤 사용자 인자를 넘겨줄 뿐이다.

## 2. 두 파일 비교 발췌

command 쪽(`commands/세법리서치.md`)은 거의 전부가 "SKILL.md를 읽고 그 절차를 따르라"는 위임이다. 그리고 끝에 `$ARGUMENTS`로 사용자 질문을 받는다.

```markdown
아래 스킬 파일을 읽고, 해당 워크플로우에 따라 질문에 답변하라.

1. Read {문서_경로}/skills/세법리서치/SKILL.md
2. 0단계: 쟁점 분석, 분야 판별, 거래처 확인
3. 1단계: 3개 Agent 병렬 호출 (반드시 단일 메시지에서 동시 호출)
4. 2단계: 교차검증   5. 3단계: 답변 작성
6. 4단계: 노션 자문기록 DB 기록   7. 5단계: 결과 보고

$ARGUMENTS        ← 사용자가 /세법리서치 뒤에 친 질문이 여기로 치환됨
```

skill 쪽(`SKILL.md`)은 frontmatter + 수백 줄의 실제 명세다. 이 본체는 트리거 조건과 절차의 디테일을 전부 담는다.

```yaml
---
name: 세법리서치
description: "... 3개 Agent 병렬 리서치 후 1.5단계 재검증 ... /세법리서치 로 호출."
version: "2.0"
---
# 세법 리서치 에이전트 (v2.0)
## 아키텍처 ... 0~5단계 + Agent A/B/C 프롬프트 템플릿 + 노션 DB 참조 ...
```

핵심 대비: command는 약 14줄, SKILL.md는 수백 줄. command는 **무엇을 부를지**만, skill은 **어떻게 할지**를 담는다.

## 3. 구조

```
.claude/
├── commands/
│   └── 세법리서치.md        # 얇은 진입점 (≈14줄)
│        └─ "SKILL.md Read 후 따르라" + $ARGUMENTS
└── skills/
    └── 세법리서치/
        └── SKILL.md          # 두꺼운 본체 (수백 줄, 실제 워크플로)

호출 흐름:
사용자: /세법리서치 "임원 퇴직금 손금산입 한도"
   ↓ command가 $ARGUMENTS = "임원 퇴직금..." 치환
   ↓ command 지시: SKILL.md를 Read
   ↓ SKILL.md 절차대로 실행 (3 Agent 병렬 → 1.5 재검증 → 노션 기록)
```

비교 관점:

| 구분 | slash command | skill |
|------|---------------|-------|
| 호출 | 사용자가 명시적으로 `/이름` 입력 | description 트리거로 자동/명시 둘 다 |
| 인자 | `$ARGUMENTS`로 사용자 입력 주입 | 대화 맥락에서 파악 |
| 내용 | 얇음 — 진입점·위임 | 두꺼움 — 전체 워크플로·리소스 |
| 재사용 | 본체를 참조만 | 여러 command가 공유하는 본체 |

## 4. 여기서 배울 패턴 3가지

1. **진입점과 본체를 분리한다.** command는 "이 기능을 시작하라"는 트리거+인자 전달만 맡고, 절차의 진실은 SKILL.md 한 곳에 둔다. 워크플로를 고칠 때 본체만 고치면 되고, command는 그대로다 — DRY(중복 제거)의 스킬 버전이다.

2. **`$ARGUMENTS`가 command의 정체성이다.** 슬래시 명령은 사용자가 `/이름 뒤에 친 텍스트`를 받는 자리(`$ARGUMENTS`)를 가진다. "임원 퇴직금 손금산입" 같은 구체 질문을 본체 워크플로에 꽂아 넣는 통로다. skill 자체는 이 치환 슬롯이 없고 대화 맥락에서 의도를 읽는다.

3. **언제 무엇을 쓰나.** 사용자가 매번 명시적으로 부르고 인자를 함께 주는 반복 작업이면 command가 깔끔하다. 모델이 맥락을 보고 "이건 세법 리서치 상황"이라 알아서 발동해야 하면 skill의 description 트리거가 맞다. 이 사례는 둘을 함께 둬서, 명시 호출(command)과 자동 트리거(skill) 양쪽을 지원한다.

## 5. 토론 질문

1. 여러분이 만들 자동화는 "사용자가 매번 명시적으로 부르는" 쪽인가요, "맥락 보고 알아서 떠야 하는" 쪽인가요? 그에 따라 command와 skill 중 무엇을 (혹은 둘 다) 만들지 어떻게 정할까요?
2. command가 본체를 "Read 후 따르라"고 위임하는 방식의 장점은 명확합니다(중복 제거). 반대로 이 방식의 약점 — 예컨대 본체가 너무 길거나 절차가 바뀔 때 — 은 무엇이고 어떻게 보완할 수 있을까요?
